# strimzi-client-go

## Project Overview

strimzi-client-go is a Go library that provides Kubernetes Custom Resource Definition (CRD) type
definitions for the [Strimzi][strimzi] Kafka operator. It enables Go programs to create, read,
update, and delete Strimzi-managed Kafka resources using strongly-typed clients rather than
unstructured Kubernetes objects.

The module is imported as `github.com/RedHatInsights/strimzi-client-go`. It targets Go 1.22+.

## Dependencies

### Runtime

- [k8s.io/api][k8s-api] v0.31.2 -- core Kubernetes API types
- [k8s.io/apimachinery][k8s-apimachinery] v0.31.2 -- API machinery (ObjectMeta, TypeMeta, runtime)
- [k8s.io/apiextensions-apiserver][k8s-apiextensions] v0.31.2 -- `apiextensions.JSON` type for
  unstructured Spec/Status fields
- [sigs.k8s.io/controller-runtime][controller-runtime] v0.19.1 -- scheme builder, typed client

### Generation tools (not runtime dependencies)

- [crd-codegen][crd-codegen] -- generates initial Go types from CRD YAML
- [controller-gen][controller-gen] -- generates `DeepCopyObject` implementations
- `convert.py` (in-repo) -- post-processing script for generated type files

## Architecture

### API groups

The library provides types for two Kubernetes API groups:

| API Group | Package | CRD Count | Key Types |
|-----------|---------|-----------|-----------|
| core.strimzi.io/v1beta2 | `apis/core.strimzi.io/v1beta2` | 1 | StrimziPodSet |
| kafka.strimzi.io/v1beta2 | `apis/kafka.strimzi.io/v1beta2` | 10 | Kafka, KafkaBridge, KafkaConnect, KafkaConnector, KafkaMirrorMaker, KafkaMirrorMaker2, KafkaNodePool, KafkaRebalance, KafkaTopic, KafkaUser |

### CRD type structure

Each CRD type follows the same pattern:

- A root struct embedding `metav1.TypeMeta` and `metav1.ObjectMeta`, with `Spec` and `Status`
  fields typed as `*apiextensions.JSON` (unstructured JSON)
- A corresponding `*List` struct wrapping `metav1.ListMeta` and a slice of the root type
- Kubebuilder marker `+kubebuilder:object:root=true` on both root and list types
- Generated `DeepCopyObject` in `zz_generated.deepcopy.go`

### Scheme registration

Each API group package exports `SchemeBuilder` and `AddToScheme`. Consumers call `AddToScheme(scheme)`
to register the group's types with their Kubernetes client scheme, enabling typed CRUD operations.

### Generated vs. hand-written code

Almost all code in `apis/` is generated. The `zz_generated.deepcopy.go` files are fully generated
by controller-gen. The CRD type files (e.g., `Kafka.go`) are generated by crd-codegen and then
lightly edited. `groupversion_info.go` files follow a standard controller-runtime pattern. The
`convert.py` script at the repo root handles post-generation formatting.

## Code Style

- **File naming:** one file per CRD, PascalCase matching the CRD kind (e.g., `KafkaBridge.go`)
- **Type naming:** PascalCase matching the CRD kind, with a `*List` companion type
- **JSON tags:** all exported fields carry `json:"name,omitempty"` tags
- **Kubebuilder markers:** `+kubebuilder:object:root=true` on root and list types
- **Unstructured fields:** Spec and Status use `*apiextensions.JSON` rather than typed structs
- **Package paths:** mirror the Kubernetes API group and version
  (e.g., `apis/kafka.strimzi.io/v1beta2`)
- **Generated files:** `zz_generated.deepcopy.go` must not be hand-edited

## Common Mistakes

1. **Hand-editing `zz_generated.deepcopy.go`.** These files are fully generated by controller-gen.
   Any manual changes will be overwritten on the next generation run. Edit the source type files
   instead, then regenerate with `controller-gen object paths=./apis/...`.

2. **Forgetting to run `convert.py` after crd-codegen.** The raw crd-codegen output requires
   post-processing. Skipping `convert.py` leaves incorrect package names, imports, or formatting
   in the generated type files.

3. **Adding strongly-typed Spec/Status fields.** The library intentionally uses
   `*apiextensions.JSON` for Spec and Status to avoid maintaining a full Go translation of the
   Strimzi CRD schema. Adding typed struct fields would create a maintenance burden that conflicts
   with the project's design.

4. **Missing `AddToScheme` call.** Consumers must call `AddToScheme` on their client's scheme
   before performing typed operations. Without registration, the client cannot serialize or
   deserialize Strimzi resource types.

5. **Creating files outside the `apis/` directory convention.** New CRD types must follow the
   existing directory structure: `apis/<api-group>/<version>/<CRDKind>.go`. Placing type files
   elsewhere breaks the controller-gen paths glob and scheme registration.
